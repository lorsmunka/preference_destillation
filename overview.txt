CODEBASE OVERVIEW
=================

Distillation pipeline: Gemma 3 4B -> small transformer (~143M params with current config).
Reddit comment classification -> deterministic JSON. KL+CE loss, reduced output vocab (525 tokens).
~12 source files, clean structure. Code is generally readable and well-organized.

INCONSISTENCIES & ODDITIES
===========================

- sleep(2) in logger._load() (line 61) and InferenceLogger.create_file() (line 60).
  No reason for either. Just slows startup.

- "radius" naming: get_training_batches_radius / get_test_batches_radius return
  (start, end) tuples. "radius" implies a center+distance, not a range.
  Should be get_training_batch_range.

- trainer.epoch_count() (line 72-73) is a method that just returns a config constant.
  Used in main.py as trainer.epoch_count(). Just use EPOCH_COUNT directly.

- Verbose prints everywhere. Every batch load, every checkpoint save, every logger
  state load prints timing info. In a 19,500 batch run, that's a lot of noise.
  BatchHandler.get_batch prints "Loading batch X..." for every single batch.

- get_batch_count() (batch_handler.py:27) calls os.listdir() every time.
  Called indirectly on every Trainer init and epoch boundary. Not cached.

- CE loss uses temperature scaling (logits/T, then *T^2) which is standard for KL
  but non-standard for CE with hard targets. Works, but unusual. Worth a comment.


LOGS vs LOGS2 ANALYSIS
======================

Setup: Both trained 4 epochs, 4,881 batches/epoch (19,524 total), identical LR schedule.
- logs/  (no annealing): kl_ratio = 1.0 throughout (100% KL, 0% CE)
- logs2/ (with annealing): kl_ratio decays 0.99 -> 0.40 via cosine schedule

Final eval metrics (epoch 3):
                        No Annealing    With Annealing    Delta
  Teacher-Forced Acc:   98.51%          98.51%            +0.00
  Student Accuracy:     92.15%          92.35%            +0.20
  Classification Acc:   84.43%          85.00%            +0.57
  KL Loss:              0.2194          0.4062            +0.187 (worse, expected)
  CE Loss:              0.5921          0.4147            -0.177 (better)

Key observations:
1. Annealing wins on the metrics that matter (student acc, classification acc)
   but the margin is small: +0.57pp classification, +0.20pp student accuracy.

2. Without annealing, CE loss stalls (0.589 -> 0.592 in last 2 epochs) because
   there's zero gradient signal from CE. With annealing, CE drops continuously.

3. No-annealing converges faster initially (epoch 0: 71.2% vs 62.2% classification)
   but annealing overtakes by epoch 1 (79.0% vs 78.5%) and the gap widens.

4. Only the annealing run ever reaches 98.8% training accuracy. Both plateau similarly
   at lower thresholds.

5. The blended loss numbers (0.165 vs 0.351) are NOT comparable - they're optimizing
   different objectives.

Verdict: Annealing is worth it. The implementation cost is ~5 lines of code
(cosine decay function + blend in loss computation). The gain is consistent
across all meaningful metrics, and it prevents CE loss stalling. For the thesis,
this is a clean result: "annealing provides measurable improvement with minimal
complexity." The improvement is modest but the added code complexity is near-zero.


FINAL THOUGHTS
==============

The codebase is solid for a thesis PoC. Main things I'd actually fix:

1. Load optimizer state in load_checkpoint. This is the only real bug that
   affects training quality.

2. Sync config.py with thesis.txt/CLAUDE.md numbers. Readers will be confused.

3. The 350-line plot function is the biggest simplification target. A loop over
   a list of (metric_name, data, color) dicts would cut it to ~80 lines.

4. Kill the sleep(2) calls. They serve no purpose.

5. Remove unused methods/lookups. Dead code is confusing.

Everything else is minor. The core training loop, model architecture, and loss
computation are clean and correct. The autoregressive distillation approach works.
