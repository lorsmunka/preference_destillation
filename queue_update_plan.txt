QUEUE-BASED PIPELINE UPGRADE PLAN (v5)
=======================================

GOAL: Multi-domain distillation pipeline. Queue-based config for data generation,
      distillation data generation (Gemma logits), and training. Two domains:
      reddit comment sentiment classification + math word problem solving.

      The end goal: queue up many training runs with different hyperparameters,
      domains, and architectures. Leave running for ~2 weeks. Come back, skim
      through neatly organized results (logs, checkpoints, eval metrics), and
      compare which configurations work best.

WORKFLOW: Claude makes changes, user reviews, user stages+commits.
Claude does NOT stage or commit. User controls git entirely.

NAMING DOCTRINE:
  The pipeline now supports multiple domains. All function names, constants,
  and variables must be domain-qualified — no generic names that could apply
  to any domain. If something is sentiment-specific, it says "reddit_sentiment".
  If it's math-specific, it says "math_word_problem". Generic names like
  "create_evaluation_prompt" or "EXAMPLE_RESPONSES" are not acceptable.
  When adding a new domain or touching existing code, check for and fix any
  ambiguous names. This prevents confusion as more domains are added.


PIPELINE OVERVIEW (3 stages)
------------------------------
1. TEXT GENERATION     -> Domain-specific input creation (Reddit extraction, math problem generation)
2. DISTILLATION GEN   -> Gemma processes inputs, autoregressively generates with logit capture per step
3. TRAINING           -> Student transformer learns from teacher logits

Key decisions made:
  - JSONL format identical for both domains: {"text": "..."}
  - No answer field needed — teacher logits ARE the ground truth
  - Semantic item categories (BAKEABLES, CONSUMABLES, ANIMALS, MANUFACTURED, etc.)
    prevent nonsense like "bakes 80 cats" or "factory makes strawberries"
  - ANIMALS excluded from ALL_ITEMS, only used via {animals} in farm/zoo templates
  - {container} = singular, {containers} = plural for grammar correctness
  - fix_singular_grammar uses regex word boundaries to avoid substring bugs
  - Division: divisor capped at range_high//2, min quotient=2 to avoid trivial results
  - Gemma 4B limitation: max 3 input variables, 1 operation before comparison

13 operation types with scaffolds (all end with ";"):
  add, sub, mul, div (2 vars)
  mul_add, mul_sub, add_mul, sub_mul, add_sub, div_add (3 vars, 2 steps)
  cmp, cmp_sub, cmp_mul (comparison, True/False output)

Number distribution (weighted random):
  20% single digits (1-9, messy)
  15% tens round (10,20,...90)
  10% tens messy (13,47,68,...)
  15% hundreds round (100,200,...900)
  10% hundreds messy (137,299,483,...)
  15% comparison tens round
  15% comparison tens messy

For hundreds+ ranges, only add/sub/div/add_sub/div_add (no mul — too large results).

DOMAIN-SPECIFIC BEHAVIOR (final):
  reddit_comment_sentiment:
    - Prompt: Utilities.create_reddit_sentiment_prompt(text)
    - Vocabulary: ~525 tokens (JSON response + whitespace + prompt + auxiliary)
    - Stop condition: generated token == "}"
    - Max generation steps: 50
    - Max sequence length: 75 (25 sentence + 50 generation)

  math_word_problem:
    - Prompt: Utilities.create_math_word_problem_prompt(text)
      Structure: MATH_WORD_PROBLEM_MASTER_PROMPT + problem_with_scaffold + "\n\nCalculations:\nA="
      The master prompt has 2 worked examples. The problem text includes the full
      scaffold with ?s (e.g. "A=?\nB=?\nC=A+B=?\nSolution: ?;"). Then "Calculations:\nA="
      forces Gemma to start filling in values immediately.
    - Vocabulary: ~31 tokens (digits 0-9, operators, variables A-G, True/False, etc.)
      Empirically discovered by running Gemma on 200 problems — 0 failures.
      Input vocab includes +2 letters beyond max scaffold variable for robustness.
    - Stop condition: generated token == ";"
    - Max generation steps: 350
    - Max sequence length: 350 (total, not additive — prompt IS the sequence)

  Key difference in sequence length semantics:
    Sentiment: additive (MAX_SENTENCE_LENGTH + MAX_GENERATION_STEPS = 25 + 50 = 75)
    Math: total (350 covers the entire prompt + generation in one number)

MATH PROMPT ENGINEERING (lessons learned):
  - Raw prompt without structure: Gemma outputs <end_of_turn> tokens and bullet formatting
  - Chat template (<start_of_turn>/<end_of_turn>): rejected, unnecessary complexity
  - Ending prompt at "\nA=": scaffold was cut off, Gemma didn't know the structure
  - SOLUTION: show full scaffold with ?s, then "\n\nCalculations:\nA=" — Gemma sees the
    structure AND is cued to start filling values. 200/200 test cases passed, 0 failures.

CONFIGURABLE PARAMETERS (all from queue, none from globals):
  Architecture:
    - hidden_dim         (wider = more capacity, more params)
    - num_layers         (deeper = more abstraction layers)
    - num_heads          (more heads = more parallel attention patterns)
    - dropout            (regularization strength)
  Training:
    - epoch_count        (how many full passes through data)
    - batch_size         (examples per gradient step)
    - learning_rate      (AdamW peak learning rate)
    - lr_warmup_ratio    (fraction of steps for linear warmup)
    - max_training_examples  (cap on data used)
    - training_test_ratio    (train/eval split)
  Distillation:
    - kl_ratio_start     (KL weight at start, 1.0 = pure KL)
    - kl_ratio_end       (KL weight at end, 0.1 = mostly CE)
    - distillation_temperature  (softens teacher logits)
  Domain-derived (automatic, not in queue):
    - output_vocab_size  (from build_vocabulary(tokenizer, domain))
    - max_seq_length     (from DOMAIN_MAX_SEQ_LENGTH[domain])

  "No annealing" = kl_ratio_start == kl_ratio_end (e.g. both 1.0).

WHY THIS WORKS:
  - Output vocab is ALREADY domain-dynamic: build_vocabulary(tokenizer, domain)
    returns ~525 tokens for sentiment, ~31 for math. Output projection auto-sizes.
  - Architecture (hidden_dim, num_layers, num_heads) is already parameterized in
    model.py constructor — just needs to receive values from queue instead of defaults.
  - Same student architecture, different domains: just change domain in queue.
    The output projection layer automatically adjusts to domain vocab size.
  - "Wider" model: increase hidden_dim (e.g. 512). "Deeper": increase num_layers.
  - "No annealing": set kl_ratio_start == kl_ratio_end (e.g. both 1.0).
  - Resumability: each run has its own state.json and temp_checkpoint.pt.
    If the process restarts, it skips completed runs and resumes the current one.
