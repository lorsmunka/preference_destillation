QUEUE-BASED PIPELINE UPGRADE PLAN
==================================

GOAL: Replace single-run config with JSON queue configs for both data generation and training.
      Enable multiple teacher models, multiple training runs, isolated outputs per model/run.


1. NEW CONFIG FILES
-------------------

A) data-generation-queue.json
[
  {
    "model_name": "google/gemma-3-4b-it",
    "max_training_examples": 500000,
    "batch_size": 32
  },
  {
    "model_name": "google/gemma-3-12b-it",
    "max_training_examples": 200000,
    "batch_size": 32
  }
]

- output_dir derived automatically: sanitize model_name -> "google_gemma-3-4b-it"
  (replace /:\<>|"?* and spaces with underscores)
- Batches land in: batches/{sanitized_model_name}/batch_1.jsonl
- Each entry is one "job". Runner processes them sequentially, skipping completed ones.

B) training-queue.json
[
  {
    "name": "baseline_kl_only",

  },

    "name": "annealed_blend",

    "data_source": "google_gemma-3-4b-it",

]

- "name" = the distillation model name, used as directory key everywhere
- "data_source" = sanitized model name, must match a dir in batches/
- All hyperparams that are currently in config.py move here (per-run overrides)

3. WHAT CHANGES IN CODE
------------------------

A) New file: shared/queue_config.py
   - load_data_generation_queue() -> list of dicts from JSON
   - load_training_queue() -> list of dicts from JSON
   - sanitize_model_name(name) -> str (replace bad chars with _)
   - TrainingRunConfig dataclass: holds all hyperparams for one training run
   - DataGenerationConfig dataclass: holds all params for one generation job

B) shared/config.py
   - Keep: get_device(), MAX_SENTENCE_LENGTH, MAX_GENERATION_STEPS, PROMPT_DELIMITER,
           MAX_SEQ_LENGTH, INFERENCE_TEMPERATURE (these are universal constants)
   - Remove: All training hyperparams (BATCH_SIZE, EPOCH_COUNT, LEARNING_RATE, etc.)
   - Remove: All directory paths (BATCHES_DIR, LOGS_DIR, CHECKPOINTS_DIR)
   - Remove: MODEL_NAME
   - These all move into the queue JSON configs

C) shared/logger.py
   - Constructor takes a base_dir parameter (e.g. "logs/training/baseline_kl_only")
   - State file, log files all write into that base_dir
   - No more global constants for file paths

D) data_generation/saving_handler.py
   - Constructor takes output_dir (e.g. "batches/google_gemma-3-4b-it")
   - mkdir that dir

E) data_generation/main.py
   - Load data-generation-queue.json
   - For each job:
     - Check if already complete (state.json in logs/generation/{sanitized_name}/)
     - If not, run generation with that model into batches/{sanitized_name}/
     - On completion, mark done in state
     - Graceful shutdown saves state, resumes at same job + same batch

F) training/batch_handler.py
   - Constructor takes batches_dir parameter (e.g. "batches/google_gemma-3-4b-it")
   - All file paths use that dir

G) training/trainer.py
   - Constructor takes TrainingRunConfig instead of reading globals
   - Checkpoints save to checkpoints/{run_name}/
   - All hyperparams come from config object, not module-level constants

H) training/main.py
   - Load training-queue.json
   - For each job:
     - Check if already complete (all epochs done in state.json)
     - Build TrainingRunConfig from JSON entry
     - Create Logger with run-specific log dir
     - Create BatchHandler with run-specific batch dir
     - Create Trainer with config
     - Run training loop
     - On completion, move to next job
     - Graceful shutdown: save temp checkpoint + state for current job, resume there


4. QUEUE COMPLETION TRACKING
-----------------------------

Each queue entry's state.json tracks completion:

Generation state.json:
{
  "status": "completed" | "in_progress",
  "processed_sentence_count": 50000,
  "successful_sentence_count": 48000,
  ...existing fields...
}

Training state.json:
{
  "status": "completed" | "in_progress",
  "current_epoch": 2,
  "current_batch": 150,
  ...existing fields...
}

Queue runner checks status before starting each job. "completed" = skip.
"in_progress" = resume. Missing state.json = start fresh.


5. GRACEFUL SHUTDOWN
---------------------

Same mechanism (% key / Ctrl+C), but scoped per-job:
- Saves temp checkpoint to checkpoints/{run_name}/temp_checkpoint.pt
- Saves state to logs/{type}/{run_name}/state.json
- Exits entire queue (not just current job)
- On restart: loads queue, finds first non-completed job, resumes from its state

8. KEY DESIGN DECISIONS
-------------------------

- Queue files are JSON (not Python) so non-programmers can edit them
- One queue file per pipeline stage (generation vs training), not one mega-config
- Training queue references data by sanitized model name, creating a clean dependency chain
- All hyperparams in the training queue JSON = full reproducibility per run
- No partial override system (no "inherit from defaults"). Each entry is self-contained.
  This is intentional: explicit > implicit, no confusion about what ran with what params.
- Sanitize function is deterministic: same model name always produces same dir name
